# 门诊医生站医嘱/病历/就诊流转说明（2026-04-10）

## 1. 文档范围

本说明覆盖门诊闭环中的三条主流程：

- 医嘱流转（药品/检查/检验）
- 病历流转（保存/签署/归档）
- 就诊流程流转（挂号到结束就诊）

目标是说明当前已落地的数据库逻辑、触发器约束、前后端职责边界。

## 2. 技术栈与落地方式

- 后端：Spring Boot + APIJSON
- 前端：React + Umi + Ant Design
- 数据库：PostgreSQL（状态机、触发器、视图、JSONB）
- 文书编辑：DCWriter（XML）
- 检查图像：MinIO（对象存储，数据库只存 URL 列表）

设计原则：

- 业务约束尽量下沉到数据库（状态机、完成前置条件、只读归档表）
- 前端负责交互与展示，状态合法性由数据库兜底
- 门诊医生站对医嘱状态仅查询展示，不直接驱动执行系统流转

## 4. 医嘱流转逻辑

### 4.1 状态机

医嘱主表 `medical_order.order_status` 使用 `order_status_enum`：

- `PENDING_RECEIVE`（待签收）
- `RECEIVED`（已签收）
- `EXECUTING`（执行中）
- `COMPLETED`（已完成）
- `CANCELLED`（已取消）

状态约束由触发器 `trg_medical_order_status_guard_bu` + 函数 `fn_medical_order_status_guard_bu` 保证：

- `PENDING_RECEIVE -> RECEIVED/CANCELLED`
- `RECEIVED -> EXECUTING/COMPLETED`
- `EXECUTING -> COMPLETED`
- `COMPLETED/CANCELLED` 为终态，禁止再修改

并自动维护：

- `status_version` 递增
- `received_time/completed_time/cancelled_time` 时间戳

### 4.2 一医嘱多明细

- 药品明细：`medical_order_drug_item`
- 检验明细：`medical_order_lab_item`
- 检查明细：`medical_order_exam`（检查部位在 `exam_part_code/exam_part`）

说明：医生开立时先写 `medical_order`，再写明细表；医生站读取时按 `order_code` 聚合展示。

### 4.3 检查与检验结果

检查结果 `exam_result`：

- 结构化：`report_jsonb`、`final_diagnosis_jsonb`
- 文书：`report_xml`（DCWriter XML）
- 图像：`image_files`（MinIO URL 数组）
- 只有 `report_status='FINAL'` 时，检查医嘱才允许完成

对应约束触发器：

- `trg_exam_order_complete_guard_bu`
- `fn_exam_order_complete_guard_bu`

检验结果 `lab_result_item`：

- 保存数值后，触发器 `trg_lab_result_item_calc_flag_biu` 自动计算 `result_flag`：
- `C`（危急）、`H`（高）、`L`（低）、`N`（正常）

### 4.4 职责边界

- 药房/检查/检验系统负责状态流转与结果录入
- 门诊医生站只查状态并展示“查看结果”入口，不驱动执行系统状态

## 5. 病历流转逻辑

### 5.1 保存与签署

门诊病历通过写入视图 `v_outpatient_emr_bundle` 统一落库，核心表：

- `outpatient_encounter_data`（就诊结构化快照）
- `emr_record`（病历主记录：`DRAFT/SIGNED` + 当前版本号）
- `emr_record_version`（病历版本内容 JSONB，含 XML）

触发器函数：

- `fn_v_outpatient_emr_bundle_u`（INSTEAD OF UPDATE）

规则：

- 每次保存新增一个 `emr_record_version`
- `sign_action=true` 时状态从 `DRAFT -> SIGNED`
- 已签署病历禁止再次修改

### 5.2 归档

历史归档表：`emr_record_history`（只允许 INSERT）

- 存储签署版本的 `emr_content` 与 `encounter_json`
- 结算前会校验必须有已签署病历

关键触发器/函数：

- `trg_emr_record_archive` / `fn_emr_record_archive`
- `trg_emr_record_history_readonly` / `fn_emr_record_history_readonly`

补充：为避免“签署瞬间历史抓到空内容”，已在 `fn_v_outpatient_emr_bundle_u` 中增加签署时同步 upsert history 的逻辑（见 hotfix 脚本）。

## 6. 就诊流程流转逻辑

### 6.1 状态机

`outpatient_visit_active.visit_status`：

- `REGISTERED`（待就诊）
- `IN_CONSULTATION`（就诊中）
- `COMPLETED`（已完成）

只允许：

- `REGISTERED -> IN_CONSULTATION -> COMPLETED`

由触发器保障：

- `trg_active_visit_status_validate`
- `trg_active_visit_finalize`

### 6.2 自动建链

挂号后自动创建就诊链路：

1. `registration` 新增
2. 触发 `trg_registration_create_active_visit` 创建 `outpatient_visit_active`
3. 触发 `trg_active_visit_create_encounter` 创建 `outpatient_encounter_data` 空 JSONB

### 6.3 结束就诊

当 `visit_status` 变为 `COMPLETED`：

- 触发器先校验已存在签署病历（`emr_record_history` 或 `emr_record.emr_status='SIGNED'`）
- 将就诊数据归档到 `outpatient_visit_history`
- 从 `outpatient_visit_active` 删除完成记录

## 7. 医生站展示口径

- 查询聚合视图：`v_outpatient_visit_detail`
- 医嘱按 `order_status` 显示状态标签
- 仅当检查/检验医嘱 `COMPLETED` 时显示“查看结果”
- 病历签署后按钮进入只读态；结束就诊前必须已签署

- 医嘱状态机：`trg_medical_order_status_guard_bu`

- 检验标志计算：`trg_lab_result_item_calc_flag_biu`
- 病历写入总入口：`trg_v_outpatient_emr_bundle_u`
- 病历归档：`trg_emr_record_archive`
- 病历历史只读：`trg_emr_record_history_readonly`
- 就诊状态机：`trg_active_visit_status_validate`
- 就诊归档结算：`trg_active_visit_finalize`
- 挂号建就诊：`trg_registration_create_active_visit`
- 就诊建 encounter：`trg_active_visit_create_encounter`
