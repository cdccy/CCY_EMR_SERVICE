# 门诊流程活跃-历史拆分改造计划

## 目标

在数据库层实现你确认的规则：

- `registration` 新增一条挂号后，自动创建一条活跃就诊记录。
- 活跃就诊记录创建后，自动创建一条 `outpatient_encounter_data`（默认 `{}`）。
- `COMPLETED`：删除活跃就诊 + JSONB 数据，并归档到历史表。
- `CANCELLED_UNSEEN`：仅删除活跃就诊 + JSONB 数据，不归档。
- 历史表只允许 `INSERT`，禁止 `UPDATE/DELETE`。

## 结构调整范围

- 复用现有表：`public.registration`、`public.outpatient_encounter_data`
- 调整现有表：`public.outpatient_visit`（重命名为 `public.outpatient_visit_active`）
- 新增表：`public.outpatient_visit_history`
- 新增类型：`public.visit_status_enum`
- 新增函数与触发器：编码生成、自动建档、状态流转、历史只读保护

## 实施步骤

1. **建状态枚举与编码函数**
  - 建 `visit_status_enum`：`REGISTERED / IN_CONSULTATION / COMPLETED / CANCELLED_UNSEEN`。
  - 建 `gen_reg_code()`、`gen_visit_code()`，生成随机且可读的业务编码；通过唯一约束兜底防重。
2. **改造挂号表编码默认值**
  - `registration.reg_code` 设置 `DEFAULT gen_reg_code()`（保留唯一约束）。
3. **活跃表改造与历史表创建**
  - `ALTER TABLE outpatient_visit RENAME TO outpatient_visit_active`。
  - `visit_status` 改为 `visit_status_enum NOT NULL DEFAULT 'REGISTERED'`。
  - 创建 `outpatient_visit_history`，字段包含活跃就诊核心字段 + `encounter_json` + `archived_at`。
  - 对历史表添加触发器：拒绝 `UPDATE/DELETE`。
4. **调整 JSONB 数据表关联**
  - `outpatient_encounter_data.visit_code` 外键从旧 `outpatient_visit` 切到 `outpatient_visit_active(visit_code)`。
  - 维持 `ON DELETE CASCADE`，保证删除活跃就诊时 JSONB 行自动清理。
5. **触发器链路实现**
  - `registration` 的 `AFTER INSERT`：自动插入 `outpatient_visit_active`。
  - `outpatient_visit_active` 的 `AFTER INSERT`：自动插入 `outpatient_encounter_data`，`encounter_json='{}'::jsonb`。
  - `outpatient_visit_active` 的 `AFTER UPDATE OF visit_status`：
    - `COMPLETED`：先写 `outpatient_visit_history` 再删除 active（触发 JSONB 级联删除）。
    - `CANCELLED_UNSEEN`：直接删除 active（触发 JSONB 级联删除）。
6. **状态流转与一致性约束**
  - 仅允许：
    - `REGISTERED -> IN_CONSULTATION`
    - `REGISTERED -> CANCELLED_UNSEEN`
    - `IN_CONSULTATION -> COMPLETED`
  - 非法流转直接 `RAISE EXCEPTION`，语句失败并回滚。
7. **测试脚本（MCP 执行）**
  - 用例 A：插入挂号，校验 active+JSONB 自动创建。
  - 用例 B：更新为 `IN_CONSULTATION`，校验仅状态变化。
  - 用例 C：更新为 `COMPLETED`，校验 active+JSONB 删除且 history 新增。
  - 用例 D：插入新挂号后更新为 `CANCELLED_UNSEEN`，校验 active+JSONB 删除且 history 不新增。
  - 用例 E：故意执行非法状态流转，验证报错并回滚。

## 关键风险与处理

- **触发器失败**：同事务自动回滚，不会出现“挂号成功但活跃/JSONB缺失”的脏状态。
- **并发冲突**：唯一索引 + 默认函数 + 幂等插入策略（必要时 `ON CONFLICT`）控制重复。
- **历史篡改风险**：历史表触发器禁止 `UPDATE/DELETE`。
- **状态混乱风险**：枚举 + 状态机触发器双重约束。
