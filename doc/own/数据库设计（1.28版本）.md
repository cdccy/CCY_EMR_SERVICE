注释：1. 为什么使用 `role_code` 关联而不是 `id`？

在你的医疗系统中，`role_code` 是**主数据属性**。使用 `code` 关联的好处在于：

- **代码可读性**：在 `sys_user` 层面，通过 `SELECT * FROM sys_user` 就能一眼看出谁是 `DOCTOR`，谁是 `ADMIN`，无需关联查询。
- **配置稳定性**：前端权限控制和 APIJSON 的 `Access` 权限配置通常基于固定编码（如 `DOCTOR`），这样即使数据库重装、自增 ID 变化，系统逻辑依然稳固。

2. 外键约束的业务语义

- **`ON UPDATE CASCADE` (级联更新)**：如果以后系统升级，将 `DOCTOR` 改为 `CLINIC_DOC`，所有关联的用户会自动同步更新，不会产生孤儿数据。
- **`ON DELETE RESTRICT` (删除限制)**：这是防止误删。如果有用户正在使用“医生”这个角色，数据库是不允许删除这个角色的，确保了临床系统的**安全性**。

3. 与 HmEditor 和医嘱系统的联动

这个 `sys_user` 表是整个流程的“发起者”：

- 当 `role_code = 'DOCTOR'` 时，系统允许其打开 **HmEditor**。
- 在 `medical_order`（医嘱表）中，记录的 `doctor_id` 实际上应该关联这个表的 `user_code`。
- **APIJSON 角色映射**：你可以通过 APIJSON 配置，让 `role_code` 直接对应到 APIJSON 的 `role` 权限，从而实现“医生只能改自己的病历，管理员能看全院数据”的逻辑。

注释：A. 挂号与患者 (fk_reg_patient_code)

- **业务含义**：不允许一个“虚假”的患者产生挂号行为。
- **CASCADE (级联)**：如果主数据中的 `patient_code` 发生修正（如由于身份证输入错误导致的号码变更），挂号记录会自动同步，保证病历不丢失。
- **RESTRICT (限制)**：如果患者已经产生了挂号记录，就不允许从 `patient` 表中物理删除该患者，只能将其 `patient_status` 设为停用。这符合**医疗数据不可篡改**的审计要求。

B. 就诊与挂号 (fk_visit_reg_code)

- **业务含义**：实现 **“挂号 $\rightarrow$ 诊疗”** 的强闭环。在门诊医生站，医生只能通过扫描或点击已有的“挂号单”来开启一份就诊单。
- **意义**：防止了医生绕过前台挂号系统自行接诊（可能涉及费用流失），确保了就诊数据的合法来源。

C. 数据冗余设计 (patient_code & doctor_code)

- **设计理由**：虽然通过 `reg_code` 可以找回患者和医生，但在 `outpatient_visit` 表中直接存放 `patient_code` 是为了**查询性能**。当你要查“某患者的所有就诊记录”时，单表查询比三表 JOIN 要快得多。
- **同步逻辑**：在插入 `outpatient_visit` 时，系统必须从对应的 `registration` 中提取这两个编码并填入。

注释：

A. 这个 JSON 里面存什么？

这里存储的是“**纯净的临床事实**”。它不带任何编辑器样式，只带标准编码（DE）。

**示例内容：**

JSON

```
{
  "DE06.00.197.00": "反复咳嗽3天，伴发热。",  -- 主诉
  "DE04.01.001.00": "38.5",                  -- 体温
  "DE05.01.024.00": "J06.900"                -- 初步诊断代码
}
```

B. JSON 是如何生成的？（业务逻辑流转）

你计划的“左边写，右边同步”过程，其 JSON 生成逻辑如下：

1. **数据收集（左侧）**：左侧页面是一个标准 HTML 表单。每个 Input 框在代码里都绑定了一个 DE 编码。
2. **实时生成**：当医生在左侧输入时，前端的状态管理（如 React 的 `useState`）会实时更新一个对象。
3. **同步渲染（右侧）**：前端监听这个对象的变化，一旦变化，立即调用 `HmEditor.setData(field, value)` 方法，将值填充到右侧病历模板的对应位置。
4. **持久化保存**：当点击“保存”时，前端直接将这个**状态对象（State Object）**作为 `encounter_json` 提交。

**`emr_template` (模板表)**：属于**主数据**。它定义了“病历长什么样”，比如内科病历需要包含主诉、现病史等段落。

**`emr_record` (实例主表)**：属于**业务主表**。它代表“某医生为某病人在某次就诊开具了一份病历”，起到枢纽作用，连接就诊、患者、医生。

**`emr_record_version` (版本表)**：属于**数据快照**。它负责存储“病历到底写了什么”。通过版本控制，实现了医疗审计所需的“可追溯性”。

**加载阶段**：医生点击接诊，系统根据就诊类型（初诊/复诊）从 `emr_template` 选出 `template_code`。

**创建阶段**：在 `emr_record` 插入一条记录，`emr_status = 'DRAFT'`。

**同步阶段**：医生在左侧输入主诉，前端通过 `DE_CODE` 将值实时推送到右侧 `HmEditor` 占位符。

**保存阶段**：点击保存，前端将 `HmEditor` 的全量 JSON 提交。系统在 `emr_record_version` 插入一条 `version_no = 1` 的记录。

**签署阶段**：医生点击签署。更新 `signed_flag = TRUE`，此时该版本被锁定。
